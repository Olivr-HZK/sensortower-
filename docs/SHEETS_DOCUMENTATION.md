# 📊 Google Sheets 工作表文档

## 📋 工作表总览

本系统共创建 **10 张**工作表，用于存储不同类型的数据和分析结果。

| # | 工作表名称 | 功能 | 主要 API 调用 |
|---|-----------|------|-------------|
| 1 | `iOS Top Charts` | iOS 榜单数据 | `/ranking`, `/category/category_history` |
| 2 | `Android Top Charts` | Android 榜单数据 | `/ranking`, `/category/category_history` |
| 3 | `📈 榜单异动` | 榜单排名变化分析 | `/ranking`, `/category/category_history` |
| 4 | `📈 起量产品` | 起量产品识别 | `/sales_report_estimates_comparison_attributes` |
| 5 | `👀 竞品动态` | 竞品公司产品统计 | `/unified/publishers/apps` |
| 6 | `📦 竞品App库` | 竞品 App 基准库 | `/unified/publishers/apps` |
| 7 | `🆕 竞品新品` | 新发现的竞品产品 | `/unified/publishers/apps` |
| 8 | `📅 历史新品` | 最近60天发布的新品 | `/apps/{appId}` |
| 9 | `📊 起量分析` | 起量产品详细分析 | `/sales_report_estimates` |
| 10 | `📊 周报汇总` | 周报汇总（不调用API） | 无（从其他表读取数据） |

**进度跟踪表**（辅助表）：
- `⏳ 扫描进度` - 历史新品扫描进度
- `⏳ 起量分析进度` - 起量分析进度

## 📄 详细工作表说明

### 1️⃣ `iOS Top Charts` - iOS 榜单数据

**创建函数**：`fetchIOSTopCharts()`

**表头结构**（7列）：
```
排名 | App ID | 应用名称 | 国家 | 榜单类型 | 平台 | 抓取日期
```

**数据来源**：
- **API 1**：`/ios/ranking` - 获取排行榜（App ID 列表）
- **API 2**：`/ios/category/category_history` - 批量获取应用名称

**调用频率**：
- 每个国家 × 每个榜单类型 = 5个国家 × 2个榜单 = **10次** `/ranking` 调用
- 应用名称：根据 App 数量，每批30个，约 **10-20次** `/category/category_history` 调用

**数据内容**：
- Top 100 应用（免费榜 + 畅销榜）
- 5个国家：美国、日本、英国、德国、印度
- 包含应用名称（带超链接到 App Store）

**更新方式**：
- 手动运行：`📊 市场监测` → `📱 榜单数据` → `iOS Top Charts`
- 一键更新：`📊 市场监测` → `🔄 一键更新所有数据`

### 3️⃣ `📈 榜单异动` - 榜单排名变化分析

**创建函数**：`analyzeRankChanges()`

**表头结构**（9列）：
```
信号 | 应用名称 | App ID | 国家 | 平台 | 本周排名 | 上周排名 | 变化 | 异动类型
```

**数据来源**：
- **API 1**：`/ranking` - 获取本周排行榜
- **API 2**：`/ranking` - 获取上周排行榜
- **API 3**：`/category/category_history` - 获取应用名称

**调用频率**：
- 本周排名：5个国家 × 2个平台 = **10次** `/ranking` 调用
- 上周排名：5个国家 × 2个平台 = **10次** `/ranking` 调用
- 应用名称：根据异动 App 数量，约 **5-10次** `/category/category_history` 调用

**数据内容**：
- 🆕 新进榜单：首次进入 Top 50
- 🚀 排名飙升：上升 ≥20 位
- 📈 排名上升：上升 ≥10 位
- 📉 排名下跌：下降 ≥20 位

**信号标识**：
- 🔴 高优先级（新进榜单、排名飙升）
- 🟡 中优先级（排名上升）
- 🟢 低优先级（排名下跌）

**更新方式**：
- 手动运行：`📊 市场监测` → `📈 分析报告` → `榜单异动分析`
- 一键更新：`📊 市场监测` → `🔄 一键更新所有数据`

---

### 4️⃣ `📈 起量产品` - 起量产品识别

**创建函数**：`identifyRisingApps()`

**表头结构**（8列）：
```
信号 | 应用名称 | App ID | 发行商 | 日均下载 | 周环比 | 国家 | 平台
```

**数据来源**：
- **API**：`/ios/sales_report_estimates_comparison_attributes` - 下载量对比数据

**调用频率**：
- **1次** API 调用（获取 Top 50 起量产品）

**数据内容**：
- 日均下载量 ≥ 5000：🔴 高优先级
- 日均下载量 ≥ 2000：🟡 中优先级
- 其他：🟢 低优先级
- 周环比变化率

**注意事项**：
- ⚠️ 需要高级 API 订阅
- ⚠️ 目前仅支持 iOS 平台
- ⚠️ 仅监控美国市场

**更新方式**：
- 手动运行：`📊 市场监测` → `📈 分析报告` → `起量产品识别`

---

### 5️⃣ `👀 竞品动态` - 竞品公司产品统计

**创建函数**：`fetchCompetitorApps()`

**表头结构**（6列）：
```
备注 | 公司名称 | Publisher ID | iOS 产品数 | Android 产品数 | 总产品数
```

**数据来源**：
- **API**：`/unified/publishers/apps` - 获取发行商所有应用

**调用频率**：
- 每个竞品公司 **1次** API 调用
- 共 **136次** API 调用（136家竞品公司）

**数据内容**：
- 136家竞品公司的产品统计
- iOS 和 Android 产品数量
- 按备注分组排序（同一备注的公司排在一起）

**更新方式**：
- 手动运行：`📊 市场监测` → `👀 竞品监控` → `竞品公司动态`

---

### 6️⃣ `📦 竞品App库` - 竞品 App 基准库

**创建函数**：`fetchNewReleases()`（首次运行）

**表头结构**（7列）：
```
App ID | 应用名称 | 公司名称 | 备注 | 平台 | 首次发现日期 | 商店链接
```

**数据来源**：
- **API**：`/unified/publishers/apps` - 获取发行商所有应用

**调用频率**：
- 首次运行：136家竞品公司 × 1次 = **136次** API 调用
- 后续运行：仅追加新品，不重新获取全部数据

**数据内容**：
- 所有竞品公司的 App 列表（iOS + Android）
- 作为基准库，用于对比发现新品
- 包含公司名称、备注、平台、首次发现日期

**更新方式**：
- 首次运行：`📊 市场监测` → `👀 竞品监控` → `竞品新品上线`
- 自动更新：每次检测新品时自动追加

**重置方式**：
- `📊 市场监测` → `👀 竞品监控` → `🗑️ 重置基准库`

---

### 7️⃣ `🆕 竞品新品` - 新发现的竞品产品

**创建函数**：`fetchNewReleases()`（后续运行）

**表头结构**（7列）：
```
发现日期 | 应用名称 | App ID | 公司名称 | 备注 | 平台 | 商店链接
```

**数据来源**：
- **API**：`/unified/publishers/apps` - 获取发行商所有应用
- **对比逻辑**：与 `📦 竞品App库` 对比，发现不在基准库中的 App

**调用频率**：
- 每次运行：136家竞品公司 × 1次 = **136次** API 调用
- 对比发现新品，追加到基准库

**数据内容**：
- 新发现的竞品产品
- 包含发现日期、公司信息、平台、商店链接

**更新方式**：
- 手动运行：`📊 市场监测` → `👀 竞品监控` → `竞品新品上线`
- 继续处理：`📊 市场监测` → `👀 竞品监控` → `▶️ 继续跑剩余竞品`

---

### 8️⃣ `📅 历史新品` - 最近60天发布的新品

**创建函数**：`scanHistoricalNewApps()`

**表头结构**（7列）：
```
发布日期 | 应用名称 | App ID | 公司名称 | 备注 | 平台 | 商店链接
```

**数据来源**：
- **API**：`/apps/{appId}` - 获取应用详情（发布日期）
- **数据源**：从 `📦 竞品App库` 读取所有 App，检查发布日期

**调用频率**：
- 每个 App **1次** API 调用
- 分批处理：每批1000个 App
- 总调用次数：取决于基准库大小（通常数千次）

**数据内容**：
- 最近60天发布的新品
- 按发布日期排序（最新的在前）
- 包含真实发布日期（从 API 获取）

**更新方式**：
- 手动运行：`📊 市场监测` → `👀 竞品监控` → `📅 扫描历史新品`
- 支持断点续传（通过 `⏳ 扫描进度` 表）

**重置方式**：
- `📊 市场监测` → `👀 竞品监控` → `🔄 重置扫描进度`

---

### 9️⃣ `📊 起量分析` - 起量产品详细分析

**创建函数**：`analyzeRisingApps()`

**表头结构**（17列）：
```
应用名称 | App ID | 公司名称 | 备注 | 平台 | 发布日期 | 
🇺🇸美国起量日 | 🇺🇸美国峰值 | 🇮🇳印度起量日 | 🇮🇳印度峰值 | 
🇯🇵日本起量日 | 🇯🇵日本峰值 | 🇬🇧英国起量日 | 🇬🇧英国峰值 |
🇩🇪德国起量日 | 🇩🇪德国峰值 | 首个起量地区
```

**数据来源**：
- **API**：`/sales_report_estimates` - 获取每日下载量数据
- **数据源**：从 `📅 历史新品` 表读取 App 列表

**调用频率**：
- 每个 App × 5个地区 = **5次** API 调用
- 分批处理：每批60个 App
- 总调用次数：历史新品数量 × 5

**数据内容**：
- 各地区的起量日期（首次突破阈值）
- 各地区的峰值下载量
- 首个起量地区判断

**阈值设置**：
- 美国/日本/英国/德国：≥ 2000
- 印度：≥ 5000

**更新方式**：
- 手动运行：`📊 市场监测` → `👀 竞品监控` → `📊 起量分析`
- 支持断点续传（通过 `⏳ 起量分析进度` 表）

**注意事项**：
- ⚠️ 需要高级 API 订阅
- ⚠️ 仅分析有起量信号的 App（任一地区起量即记录）

## 🔄 数据流向

```
API 数据获取
    ↓
工作表存储
    ↓
数据分析/汇总
    ↓
周报汇总 / Google Doc 导出
```

### 主要数据流

1. **榜单数据流**：
   ```
   /ranking → iOS/Android Top Charts → 榜单异动 → 周报汇总
   ```

2. **竞品监控流**：
   ```
   /unified/publishers/apps → 竞品App库 → 竞品新品
   ```

3. **新品分析流**：
   ```
   竞品App库 → /apps/{appId} → 历史新品 → /sales_report_estimates → 起量分析
   ```

---

## 💡 使用建议

1. **定期更新**：建议每周运行一次 `🔄 一键更新所有数据`
2. **竞品监控**：建议每天运行一次 `竞品新品上线`
3. **历史扫描**：建议每月运行一次（数据量大，耗时较长）
4. **起量分析**：建议在历史新品扫描完成后运行
